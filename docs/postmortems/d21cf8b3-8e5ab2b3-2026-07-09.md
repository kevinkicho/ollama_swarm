## Cross-cutting: Agent 1 stuck “thinking” after run end

### Symptom

After `swarm_state` transitioned to `stopped` / `completed`, Agent 1 (planner/lead) could remain in **thinking** in the sidebar while streaming had cleared.

### Root cause

1. `setPhase` on terminal phases cleared `streaming` / `streamingMeta` but did not reset agent status from `thinking` / `retrying` → `ready`.
2. `shouldDropTerminalGuardedEvent` could drop a final `agent_state: ready` event on hydrated/historical views, leaving the last live state stuck.

### Fix

`web/src/state/store.ts` — terminal `setPhase` now resets any agent still in `thinking` or `retrying` to `ready`.

## Lessons

1. **Transport failures are not semantic stalls** — stuck-cycle counters must distinguish quota/429 from “same failure signature” on real code attempts.
2. **Stop reason is a product surface** — `completed` with 0/N criteria met erodes trust; gate on `completionDetail` + unmet criteria, not commits alone.
3. **Second-hand help must be event-routed** — transcript system lines are not enough; operators need a control panel and agent-injected hints on the next turn.
4. **Skip is a failure mode** — replanner “already done” skips need disk grounding when criteria remain unmet; otherwise skip storms masquerade as progress.
